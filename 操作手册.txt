  一、日常更新流程（4 步）

  # 1. 新建文章（自动套用你的分类/标签骨架）
  npx hexo new post "文章标题"

  # 2. 写。用 VS Code 等打开 source/_posts/文章标题.md，填好正文和 front-matter

  # 3. 本地预览（强烈建议养成习惯）——浏览器开 http://localhost:4000 实时看效果
  npx hexo server
  #    看完按 Ctrl+C 停掉

  # 4. 满意了就推送，约 1 分钟后自动上线
  git add -A
  git commit -m "新增：文章标题"
  git push

  没写完的草稿别直接放 _posts（会上线）：
  npx hexo new draft "半成品标题"     # 存到 source/_drafts/，不会发布
  npx hexo publish "半成品标题"        # 定稿后一键转正到 _posts/

  二、治理"兴趣多、很混乱"——这是核心

  你之前乱，是因为结构定太早、太细。记住一个反直觉的原则：

  ▎ 标签先行，分类后长。 新内容先只打
  ▎ tags（随便贴关键词），别急着开新分类。等某个主题攒到一定数量、自然成簇了，再把它"提拔"成分类。

  配一条硬规则防止分类爆炸：

  ▎ 「3 篇定律」：只有当某主题真的写了 ≥3 篇、形成一个簇，才给它建一个（子）分类。不到 3 篇，一律用 tags 装着。

  顶层分类要少而稳（你的人生大领域，长期就 4–6 个，很少变）。现在你只有计算机，以后兴趣扩展时，蓝图大概长这样：

  技术笔记        ← 现在的（渗透/Web/网络…二级随簇生长）
    ├─ 渗透测试
    ├─ Web安全
    └─ 网络基础
  数学            ← 等数学笔记攒到 3 篇再开
  硬件 / 电子      ← 同理
  AI / 机器学习
  读书笔记         ← 跨领域的书评/摘录
  杂学            ← 还没成簇的零散兴趣，先都丢这儿 + 打 tags

  判断流程（每写一篇问自己）：
  1. 能塞进现有某个顶层抽屉吗？→ 能就放进去，二级看 tags 够不够细
  2. 是个全新大领域、且预感会持续写？→ 才考虑开新顶层分类
  3. 拿不准 / 只是一时兴起？→ 丢进「杂学」+ 多打 tags，以后成簇了再迁移

  这样你想写啥写啥，结构却始终清爽——乱的部分都被 tags 吸收了，分类树永远只有几个稳定抽屉。
