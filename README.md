# 博客维护手册

个人博客（Hexo + NexT），部署在 GitHub Pages：<https://inkroms.github.io>
推送到 `main` 分支后由 GitHub Actions 自动构建上线，无需手动操作服务器。

---

## 一、写一篇并发布（日常流程）

```bash
# 1. 新建文章（自动套用分类/标签骨架）
npx hexo new post "文章标题"

# 2. 编辑 source/_posts/文章标题.md，填正文和 front-matter

# 3. 本地预览，浏览器开 http://localhost:4000 实时看效果
npx hexo server          # 看完按 Ctrl+C 退出

# 4. 推送，约 1 分钟后自动上线
git add -A
git commit -m "新增：文章标题"
git push
```

### 草稿（没写完别上线）
```bash
npx hexo new draft "标题"     # 存到 source/_drafts/，不会发布
npx hexo publish "标题"        # 定稿后转正到 source/_posts/
```

## 二、分类与标签规则

| | 作用 | 数量 | 怎么填 |
|---|---|---|---|
| **categories 分类** | 固定的"抽屉"，回答"属于哪类" | 每篇一个路径 | 顶层少而稳，二级写领域 |
| **tags 标签** | 自由关键词，回答"涉及什么" | 每篇可多个 | 工具名、技术点、书名…随手贴 |

### 核心原则
1. **标签先行，分类后长**：新内容先只打 tags，别急着开新分类。
2. **「3 篇定律」**：某主题写满 ≥3 篇、成簇了，才给它建（子）分类；不到 3 篇用 tags 装着。
3. **顶层分类少而稳**（长期 4–6 个）。

### 当前分类蓝图
```
技术笔记
  ├─ 渗透测试
  ├─ Web安全
  └─ 网络基础
杂学          ← 还没成簇的零散兴趣，先丢这里 + 多打 tags，以后成簇再迁移
（数学 / 硬件 / AI / 读书笔记… 等各自攒满 3 篇再开顶层分类）
```

### 每写一篇问自己
1. 能塞进现有抽屉吗？→ 能就放进去
2. 是全新大领域且会持续写？→ 才开新顶层分类
3. 拿不准 / 一时兴起？→ 丢进「杂学」+ 多打 tags

> 口诀：**领域用分类，书名/工具/具体技术用标签。** 想不好放哪 → 那是分类；还涉及啥 → 那些是标签。

## 三、注意事项
- 仓库**公开**：别把密钥、私人/隐私内容写进文章或提交历史。私人随笔本地另存，不发布。
- 图片注意去除 EXIF（手机照片含 GPS）。
- `db.json` / `public/` / `node_modules/` / `.claude/` 已被 `.gitignore` 忽略，不会上传。
- 提交身份已设为 noreply 邮箱，不暴露真实邮箱。
